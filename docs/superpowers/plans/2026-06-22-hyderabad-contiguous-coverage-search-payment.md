# Hyderabad Contiguous Coverage, Search, and Paid-Access Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make every marketable point inside the agreed Hyderabad investment-market boundary selectable and resolvable, make locality/full-address search land on the correct polygon and analysis, and automatically restore verified Rs 99 access across sessions and devices.

**Architecture:** Replace the current floating, hand-sized locality polygons with a provenance-tracked coverage model: sourced administrative/locality boundaries where available, explicit special-use masks, and generated fallback market cells only where no authoritative locality boundary exists. Route both map clicks and text/address searches through one backend-owned resolver contract. Store payment facts separately from anonymous sessions and grant access only from signature-verified Razorpay events or server-to-server payment verification.

**Tech Stack:** FastAPI, Pydantic, GeoPandas/Shapely/PyProj, React 19, TypeScript, MapLibre, SQLite locally with a production Postgres-compatible schema, Razorpay webhooks/API, and HTTPX-based geocoding.

---

## Requested rollout order

1. **Contiguous Hyderabad market coverage** — airport/south, LB Nagar/east, Rangareddy, Sangareddy, Isnapur, west/northwest, and every marketable point inside the agreed boundary.
2. **Locality and complete-address search** — exact/alias/typo search first, then server-side address geocoding; select the actual resolved polygon instead of silently substituting the nearest supported locality.
3. **Automatic paid-access restoration** — a verified payment follows the buyer's verified identity across login, refresh, browser, and device changes.

One security-only guardrail precedes those phases: disable unverified client self-confirmation. It does not reorder feature delivery; it closes an existing free-unlock path before further work.

## Current-state findings

- Hyderabad contains 240 locality polygons and 240 catalog cards, but the production check asserts a fixed count rather than geographic completeness (`frontend/scripts/check-hyderabad-production.mjs:28-33`).
- The shapes are synthetic centroid buffers generated by scripts such as `scripts/expand_localities.ps1:130-139`, which explains the visible islands and holes in the supplied screenshot.
- `data/cities/hyderabad/localities.json` has 240 records while `data/cities/hyderabad/aliases.json` has only 200 keys. The 40 missing alias entries include Sangareddy, Sadasivpet, Isnapur-adjacent west-corridor records, and other recently added markets.
- Map rendering directly converts each catalog polygon into an independent GeoJSON feature (`frontend/src/components/map/MapView.tsx:241-264`); there is no market boundary, gap layer, water mask, or special-use-area layer.
- Text search only performs case-insensitive exact/prefix/substring matching against the displayed catalog name (`frontend/src/pages/Home.tsx:190-207`). Address text, aliases, misspellings, landmarks, and PIN codes do not enter a geocoder.
- The backend already geocodes brochure/map-link text with Nominatim (`backend/app/api/routes/utils.py`), but it does not expose a general address-search endpoint. Public Nominatim must not be used for client-side autocomplete; its published policy is at <https://operations.osmfoundation.org/policies/nominatim/>.
- Coordinate resolution already supports polygon, nearby, cluster, and regional tiers (`backend/app/services/location_resolver.py:146-388`), but an uncovered point can inherit a nearby market and score instead of preserving the searched place as the primary identity.
- Razorpay webhook signatures are verified from the raw request body (`backend/app/api/routes/leads.py:83-115`), which is the correct trust boundary.
- The client-visible self-confirm route changes a pending lead to paid without provider verification (`backend/app/services/custom_report_leads.py:161-210`). Payment-ID recovery similarly accepts any `pay_`-prefixed string (`backend/app/services/custom_report_leads.py:214-295`). Both can grant entitlements and must not remain production unlock paths.
- Entitlements are keyed to an anonymous `user_id` (`backend/app/services/entitlements_store.py:80-153`), while payment leads are JSONL records. A returning buyer with a new anonymous token is therefore not automatically linked to the earlier payment.

## Definition of “complete”

Completion is geographic and behavioral, not a locality-count target.

- A committed `marketBoundary` defines the Hyderabad investment-market scope and contains at minimum the requested anchors: Sadasivpet, Sangareddy, Isnapur, the western corridor, Medchal/northern corridor, LB Nagar/eastern corridor, RGIA/Shamshabad, Maheshwaram, and Farooqnagar/Shadnagar.
- At least **99.5% of marketable land area** inside `marketBoundary` belongs to exactly one selectable coverage polygon.
- No unexplained uncovered marketable-land gap may exceed **0.25 km²**.
- Polygon overlaps may not exceed **0.1% of marketable land area**. Shared borders are allowed; overlapping selectable interiors are not.
- Water bodies, airport operational land, defence land, forests/protected land, and other non-marketable special-use areas are represented by named masks. They are not disguised as residential localities and do not count as coverage holes.
- Every selectable polygon has a unique slug, display name, centroid, geometry provenance, boundary confidence, analysis confidence, aliases, and a catalog-backed result. Estimated/corridor-derived scores are visibly labelled `estimated`; they are never presented as verified exact-market measurements.
- Every requested anchor and the search regression corpus resolves to the expected polygon and never silently changes the primary place name to a nearby locality.

## Risks and mitigations

- **False boundary precision:** Sourced polygons may be outdated or incomplete. Preserve source IDs/dates and label generated fallback cells as broad rather than administrative truth.
- **Fake score precision:** Contiguous geometry does not create verified market intelligence. Carry `scorePrecision` and `scoreSourceSlug` through API/UI and keep estimated analysis visibly distinct.
- **Geometry size/performance:** Hundreds or thousands of detailed polygons can slow the browser. Simplify only display geometry by zoom while keeping unsimplified canonical geometry for backend containment; measure bundle/GeoJSON size and map interaction latency.
- **Ambiguous place names/PIN codes:** Rank within the selected market boundary and show disambiguation instead of auto-selecting an uncertain result.
- **Geocoder availability/policy:** Cache submitted searches, keep local autocomplete independent, enforce timeouts, and support a configurable provider rather than hard-coding public Nominatim behavior.
- **Payment fraud/replay:** Only verified provider facts grant access; use raw-body signature verification, unique event/payment IDs, amount/package checks, and transactional idempotency.
- **Identity collision:** Restore by OTP-verified normalized email, not raw form entry. Phone-only restoration requires separately verified phone ownership and is outside this plan.
- **Local-to-production persistence gap:** Do not launch cross-device restoration on ephemeral SQLite/JSONL storage. Use durable production storage with migration and rollback evidence.

- Phase 1 ships only when spatial metrics and visual QA both pass; a larger polygon count alone is insufficient.
- Phase 2 ships only when every search-regression fixture selects the expected polygon and preserves precision labels.
- Phase 3 ships only when fabricated-payment tests fail closed, duplicate webhooks are idempotent, and a new-device OTP flow restores a real verified test payment.
- The product may use “Hyderabad flagship” language only after Phase 1 and Phase 2 gates pass. Payment restoration is a commerce reliability gate, not a geographic flagship gate.

## Remaining decisions during execution

These are evidence decisions, not open product-scope questions:

- Choose the best legally usable boundary source after comparing authoritative coverage, freshness, downloadability, and licence.
- Decide which special-use masks are reliable enough for the first release; any unavailable mask must remain an explicitly reported gap, not be absorbed silently into a residential polygon.
- Select the production geocoding provider based on licence, SLA, India address quality, and cost after the provider adapter and corpus exist.
- Select production Postgres/Supabase deployment details while preserving the payment schema and trust rules defined above.
